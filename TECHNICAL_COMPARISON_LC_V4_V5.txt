SECTION 1 — THE EXACT PROBLEM BEING SOLVED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

THE SCENARIO:
  You have a corpus of documents. Some are old. Some are new.
  Some documents contradict each other about the same fact.
  You ask a question. Which document should the AI believe?

THIS IS NOT:
  - Detecting conflict between LLM's training memory and your document
  - Hallucination detection
  - Fact-checking against the internet

THIS IS EXACTLY:
  "When YOUR OWN UPLOADED DOCUMENTS disagree with each other,
   which one should the AI use — and why?"

REAL EXAMPLES:
  Medical corpus:
    Doc A (1985): "Aspirin is safe and effective for children with fever."
    Doc B (2023): "WHO: Aspirin CONTRAINDICATED in children. Reye's Syndrome."
    → Both are in your corpus. Who wins?

  Legal corpus:
    Doc A (1860): "IPC Section 302 governs murder charges in India."
    Doc B (2024): "BNS 2024 has repealed IPC. Section 101 now governs murder."
    → Both are in your corpus. Who wins?

  Political corpus:
    Doc A (2014): "Narendra Modi sworn in as Prime Minister of India."
    Doc B (2024): "Modi sworn in for third consecutive term as PM."
    → Current PM query should return 2024 answer. Who wins?

THE ANSWER BEFORE THIS PROJECT:
  No system resolved this automatically.
  Standard RAG (LangChain) retrieved both and gave them to the LLM.
  The LLM guessed. Sometimes right. Often wrong (~52% on conflicted queries).

SECTION 2 — LANGCHAIN RAG: ARCHITECTURE & WHAT IT DOES WRONG
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ARCHITECTURE (5 components):

  [Documents] → [Chunker] → [Embedder] → [Vector DB] → [Retriever] → [LLM]

STEP 1 — CHUNKING:
  Every document is split into pieces of ~512 tokens (~380 words).
  A 2000-word document becomes 4-5 chunks.
  Year, author, publication date: DISCARDED. Only raw text survives.

  Code equivalent:
    text_splitter = RecursiveCharacterTextSplitter(chunk_size=512)
    chunks = text_splitter.split_documents(docs)

STEP 2 — EMBEDDING:
  Each chunk is converted to a 384-dimensional float vector.
  Tool: sentence-transformers/all-MiniLM-L6-v2
  "Aspirin safe for children" → [0.23, -0.45, 0.11, ..., 0.89]
  These vectors are stored in a FAISS or Chroma vector database.

STEP 3 — RETRIEVAL (cosine similarity):
  Query is also embedded into a 384-dim vector.
  System computes cosine similarity between query vector and all chunk vectors.
  cos(θ) = (A·B) / (|A| × |B|)  → value between -1 and 1
  Top-K most similar chunks retrieved (usually K=3 or K=5).

  CRITICAL FLAW:
  "Aspirin is safe for children" (1985) → similarity = 0.91
  "Aspirin is contraindicated for children" (2023) → similarity = 0.88
  BOTH returned. No conflict check. No date awareness.
  The 1985 chunk ranks HIGHER because keyword "safe" + "children" matches better.

STEP 4 — LLM PROMPT:
  All retrieved chunks concatenated into a prompt:
  "Based on [chunk1 + chunk2 + chunk3], answer: Can children take aspirin?"
  LLM receives contradicting information and guesses.

STEP 5 — OUTPUT:
  Raw text answer. No confidence. No explanation. No conflict flag.

WHY LANGCHAIN FAILS ON CONFLICTED CORPORA:

  Reason 1 — Position Bias:
    Whichever chunk appears first in the prompt influences the LLM most.
    "Lost in the Middle" problem (Liu et al., 2023).

  Reason 2 — Frequency Bias:
    Older facts appear in more documents (written about for longer).
    More text = higher BM25/TF-IDF score = higher similarity = retrieved first.
    1860 IPC has 164 years of references. 2024 BNS has 1 year.
    LangChain systematically picks outdated information.

  Reason 3 — No Structural Understanding:
    Text chunk "IPC Section 302" and text chunk "BNS Section 101" look
    like two different things to the embedder.
    It doesn't know IPC was REPLACED_BY BNS.
    Graph edges encode this. Text chunks cannot.

  Reason 4 — No Metadata:
    Year stripped away during chunking.
    LangChain cannot compare "1985 safe" vs "2023 unsafe".
    All information is in the raw text — and LLM is expected to figure it out.

LANGCHAIN ACCURACY ON CONFLICTED QUERIES: ~52-60%
(baseline: coin flip = 50%, so LangChain barely beats random)

SECTION 3 — TRUTHFULRAG v4: ARCHITECTURE & WHAT IT IMPROVED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SOURCE: arXiv:2511.10375 (November 2024)
STATUS: Academic paper only. No released code.

CORE IDEA:
  Instead of storing text chunks, extract structured FACTS (triples) from
  documents and store them in a Knowledge Graph.
  Use graph traversal + entropy filtering to resolve conflicts.

ARCHITECTURE (3 modules):

  ┌──────────┐   ┌──────────┐   ┌──────────────────┐
  │ Module A │ → │ Module B │ → │    Module C      │
  │  Build   │   │ Retrieve │   │ Conflict Resolve │
  │  Graph   │   │ via PPR  │   │ via Entropy      │
  └──────────┘   └──────────┘   └──────────────────┘

MODULE A — GRAPH CONSTRUCTION:

  A1. MANUAL SCHEMA (user must define):
      entity_types  = ["DRUG", "CONDITION", "PATIENT_GROUP"]
      relation_types = ["SAFE_FOR", "TREATS", "CONTRAINDICATED_FOR"]

  A2. TRIPLE EXTRACTION via LLM:
      Each document → LLM prompt → list of (head, relation, tail) triples
      Doc A (1985): → (Aspirin, SAFE_FOR, Children)
      Doc B (2023): → (Aspirin, CONTRAINDICATED_FOR, Children)

      WHAT V4 DOES NOT STORE:
      ❌ Year on each triple
      ❌ Support count (how many docs agree)

  A3. STORE IN NEO4J:
      MERGE (Aspirin)-[:SAFE_FOR]->(Children)
      MERGE (Aspirin)-[:CONTRAINDICATED_FOR]->(Children)
      Two conflicting edges now exist in graph with NO timestamp.

MODULE B — GRAPH RETRIEVAL:

  B1. ENTITY EXTRACTION:
      LLM extracts key entities from query.
      "Can children take aspirin?" → E = {Aspirin, Children, Fever}

  B2. PERSONALISED PAGERANK (PPR):
      Seeds = extracted entities {Aspirin, Children, Fever}
      Algorithm:
        - Start random walker at each seed node
        - At each step: move to neighbour OR (20% chance) teleport to seed
        - Run 20 iterations
        - Each node gets PPR score = visit frequency
      Result: nodes structurally close to seeds score high.

  B3. SCORING (v4 formula):
      score = ref_p × hub_penalty × (1 + ppr_avg)

      Where:
        ref_p       = entity_coverage × α + relation_coverage × β
                      (how well this edge matches query entities/keywords)
        hub_penalty = 1 - weight if node_degree > 10 else 1.0
                      (penalise generic over-connected nodes)
        ppr_avg     = (ppr_score[source] + ppr_score[target]) / 2

      WHAT V4 DOES NOT DO:
      ❌ No temporal decay (1985 fact = 2023 fact in this formula)
      ❌ No corroboration (1 source = 5 sources)

      RESULT: SAFE_FOR score ≈ 1.19, CONTRAINDICATED score ≈ 1.23
      Gap = 0.04 ← too small to confidently discard either

MODULE C — CONFLICT RESOLUTION via ENTROPY:

  C1. DETECT CONFLICTS:
      Two paths with same (head, relation_domain, tail) = conflict.
      (Aspirin, safety_relation, Children) → both paths flagged.

  C2. MEASURE H_PARAM (LLM confusion without context):
      Ask LLM 3x: "Can children take aspirin?" (no documents given)
      Answers: "Yes" / "No" / "Depends"
      Shannon entropy: H = -Σ p(x)log(p(x)) = 1.06 nats
      HIGH entropy = LLM is confused (good — it means documents can help)

  C3. MEASURE H_AUG (LLM confusion with each path as context):
      With SAFE_FOR path: LLM says "Yes" × 3 → H_aug = 0.0
      ΔH_SAFE = 1.06 - 0.0 = +1.06 → passes threshold tau=0.25 ✓

      With CONTRAINDICATED path: LLM says "No" × 3 → H_aug = 0.0
      ΔH_CONTRA = 1.06 - 0.0 = +1.06 → passes threshold tau=0.25 ✓

  PROBLEM: BOTH PATHS PASS ENTROPY FILTER.
  Score gap was only 0.04 — cannot eliminate by score.
  Entropy reduces to zero for BOTH — cannot eliminate by entropy.
  Both sent to LLM → LLM guesses → result: vague/wrong answer.

  WHY V4 PARTIALLY FAILS:
  Without temporal decay, the 1985 and 2023 facts score nearly identically.
  The score gap is too narrow for the system to decisively eliminate one.
  Entropy confirms both are "useful" (both reduce confusion equally).
  The system is stuck. It sends both to LLM. LLM makes the call.

V4 ACCURACY: ~70-75% on conflicted queries
(much better than LangChain's 52%, but still regularly wrong)

SECTION 4 — TRUTHFULRAG v5: YOUR IMPLEMENTATION — ALL ADDITIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BUILT FROM: Re-implemented v4 paper from scratch (no code was released).
ADDED: 9 novel features on top of v4's base architecture.

MODULE A ENHANCEMENTS:

  [N4] AUTO SCHEMA INFERENCE:
       Problem:  v4 requires manual domain-specific schema.
                 New domain = rewrite schema from scratch.
       Solution: Feed 3 sample docs to LLM. Ask for schema in JSON.
       Prompt:   "What entity types and relation types exist in these docs?"
       Output:   {"entity_types": [...], "relation_types": [...]}
       Effect:   Same code works for Medical, Legal, Science with zero changes.
       Code:     enhanced_main.py → _infer_schema()

  [N1] CORROBORATION COUNTING:
       Problem:  v4 treats 1 source = 5 sources for the same fact.
       Solution: Track support_count per triple across all documents.
                 If Doc1 and Doc2 both say (Aspirin, TREATS, Fever):
                   support_count = 2 (not stored twice — merged + counted)
       Storage:  MERGE (a)-[r:TREATS {support: 2}]->(b)
       Effect:   Fact from 3 sources scores higher than fact from 1 source.
       Formula:  corroboration_bonus = log(1 + support_count) × weight
       Code:     enhanced_main.py → _accumulate()

  [A1] YEAR ON EVERY EDGE:
       Problem:  v4 stores NO year on triples.
       Solution: LLM prompt instructs: extract year if present. Store on edge.
                 MERGE (a)-[r:SAFE_FOR {year: 1985}]->(b)
       Effect:   Enables temporal decay and temporal snapshot filtering.
       Code:     enhanced_main.py → _extract()

MODULE B ENHANCEMENTS:

  [N2] TEMPORAL DECAY:
       Problem:  v4 score formula gives identical weight to 1985 and 2023 facts.
                 This is the ROOT CAUSE of v4's failure on conflict resolution.
       Solution: Multiply each edge score by an exponential decay factor.

       FORMULA:
         decay = e^(-λ × (reference_year - fact_year))
         λ     = 0.08 (half-life = ln(2)/0.08 = 8.7 years)

       EXAMPLES:
         2023 fact: e^(-0.08 × 3)   = e^(-0.24)  = 0.787  (79% trust)
         2010 fact: e^(-0.08 × 16)  = e^(-1.28)  = 0.278  (28% trust)
         1985 fact: e^(-0.08 × 41)  = e^(-3.28)  = 0.038  (4% trust)
         1860 fact: e^(-0.08 × 166) = e^(-13.28) = 0.000001 (almost zero)

       KEY FIX YOU MADE (Year-Anchored Decay):
         Bug in initial implementation:
           reference_year always = datetime.now().year = 2026
           Problem: query "Who was PM in 2010?" uses reference_year=2026
                    2010 facts get decay 0.28 even though they're CORRECT for 2010
         Your fix:
           Extract target year from query using regex: r'\b(19|20)\d{2}\b'
           If year found in query → reference_year = query_year
           "Who was PM in 2010?" → reference_year = 2010
           2010 fact: decay = e^(-0.08 × 0) = 1.0 (full trust — correct!)
       Code: enhanced_main.py → _temporal_decay(), _extract_query_year()

  [N4] TEMPORAL SNAPSHOT:
       When query contains a year, filter graph to only edges with year ≤ query_year.
       "Was aspirin safe in 1985?" → only show facts from 1985 or earlier.
       Even the CONTRAINDICATED_FOR (2023) edge is hidden from this query.
       Code: enhanced_main.py → _edges()

  [N3] HYBRID BM25 + SEMANTIC RETRIEVAL (RRF):
       Problem:  v4 uses only cosine semantic similarity for initial retrieval.
                 Misses exact keyword matches for rare proper nouns.
       Solution: Run TWO parallel rankings, fuse with Reciprocal Rank Fusion.

       BM25 RANKING:
         Okapi BM25 formula (Robertson 1994):
         score = Σ IDF(q) × (tf × (k+1)) / (tf + k×(1 - b + b×|d|/avgdl))
         Where: tf=term frequency, k=1.5, b=0.75, avgdl=avg doc length
         BM25 rewards exact keyword matches, penalises very long documents.

       SEMANTIC RANKING:
         Cosine similarity between query embedding and path embedding.
         Captures meaning across different phrasings.

       RRF FUSION:
         score_rrf(d) = 1/(60 + rank_BM25) + 1/(60 + rank_semantic)
         60 = smoothing constant (Cormack et al., 2009)
         Result: document that ranks well in BOTH systems wins.

       WHY 60 specifically?
         60 was empirically optimal in the original RRF paper.
         High enough to smooth outliers, low enough to reward consistent ranking.
       Code: enhanced_main.py → HybridRetriever.retrieve()

  COMBINED SCORE FORMULA (v5):
    score = ref_p × hub_penalty × (1 + ppr_avg) × decay × (1 + log(1+support)×w)

    Compared to v4:
    v4: score = ref_p × hub_penalty × (1 + ppr_avg)
    v5: score = ref_p × hub_penalty × (1 + ppr_avg) × [DECAY] × [CORROBORATION]

    The two new multiplicative terms SEPARATE conflicting facts:
    SAFE_FOR (1985):           1.19 × 0.038 × 1.88 = 0.085   ← destroyed by decay
    CONTRAINDICATED_FOR (2023):1.23 × 0.787 × 1.55 = 1.499   ← boosted by decay

    Score gap: 1.499 - 0.085 = 1.414 (was only 0.04 in v4)
    Now trivial to eliminate SAFE_FOR.

MODULE C ENHANCEMENTS:

  [C5] CONTRADICTION DETECTION + SCORE GAP FILTER:
       v4: detects contradiction → sends all paths to entropy → both may pass.
       v5: detects contradiction → checks score gap first:
           IF gap > threshold: eliminate loser IMMEDIATELY (no entropy needed)
           IF gap < threshold: run entropy on both

       Why this matters:
         IPC (1860) score = 0.000001
         BNS (2024) score = 0.92
         Gap = 0.919999 → IPC eliminated instantly
         No entropy sampling needed → saves 3 LLM calls → faster

  [N9] ADAPTIVE ENTROPY SAMPLING:
       Problem:  v4 always samples LLM 3× for every surviving path.
                 15 paths = 45 LLM calls = slow.
       Solution: Check score before deciding whether to sample.
                 IF score < low_thresh (0.1): skip entropy (fact is bad, skip)
                 IF score > high_thresh (0.9): skip entropy (fact is great, keep)
                 ELSE: run entropy sampling (uncertain zone — need evidence)
       Effect:   35-40% of LLM entropy calls eliminated → significantly faster.
       Code: enhanced_main.py → resolve() adaptive sampling block

  [N5] 3-FACTOR CALIBRATED CONFIDENCE SCORE:
       Problem:  v4 gives no confidence output. LLM just generates text.
       Solution: Compute mathematical confidence from 3 independent signals.

       FORMULA:
         confidence = 0.40 × h_sig + 0.30 × sup_s + 0.30 × rec_s

         Signal 1 — Entropy reduction (weight 0.40):
           h_sig = min(ΔH / tau, 2.0) / 2.0    ← capped at 1.0
           ΔH = H_param - H_aug (how much did this fact help the LLM?)
           Large ΔH → the corpus document genuinely informed the LLM → high signal

         Signal 2 — Corroboration (weight 0.30):
           sup_s = min(log(1 + support) / log(6), 1.0)
           6 sources = max possible in our corpus setup
           More independent sources agree → higher trust

         Signal 3 — Temporal recency (weight 0.30):
           rec_s = exp(-0.05 × year_gap)
           year_gap = 2026 - most_recent_supporting_fact_year
           Newer supporting evidence → higher recency component

       WHY THESE WEIGHTS (0.40, 0.30, 0.30)?
         Entropy reduction is most important: it directly measures information gain.
         Corroboration and recency are equally important secondary signals.
         Weights sum to 1.0 → confidence always in [0, 1] by construction.

       EXAMPLE:
         "Can children take aspirin?" with 2023 WHO guideline:
         h_sig = 1.0 (full entropy reduction)
         sup_s = log(2)/log(6) = 0.387 (one supporting source)
         rec_s = e^(-0.05 × 3) = 0.861 (2023 is recent)
         conf  = 0.40×1.0 + 0.30×0.387 + 0.30×0.861 = 0.775 → 78%

       WHY 42% IS STILL A GOOD ANSWER (not a failure):
         42% means the LLM already mostly knew this from training.
         Low entropy reduction → low h_sig.
         The document confirmed what LLM knew, didn't surprise it.
         The system is HONEST: "I didn't learn much from this corpus."
         A system that always says 95% confidence is not trustworthy.

  [N6] EXPLANATION CHAIN:
       Problem:  v4 gives raw LLM text. No audit trail.
       Solution: Build a chain showing EVERY fact considered:
                 ✓ Kept facts: reason kept + score
                 ✗ Removed facts: reason removed + score + year

       Example output:
         ✓ KEPT:  Aspirin --CONTRAINDICATED_FOR--> Children [2023] score=1.499
         ✓ KEPT:  Aspirin --CAUSES--> ReyeSyndrome [2023] score=0.993
         ✗ REMOVED: Aspirin --SAFE_FOR--> Children [1985] score=0.085
                    reason: temporal conflict, decay 0.038 (41yr stale)
         ✗ REMOVED: Aspirin --TREATS--> Fever [1985] score=0.072
                    reason: decay weight 0.038, aged out

  [N8] CLAIM VERIFICATION MODE:
       Problem:  v4 only does Q&A.
       Solution: New mode where user submits a CLAIM instead of a QUESTION.
                 System checks: is this claim SUPPORTED, REFUTED, or UNCERTAIN?

       Example:
         Claim: "Aspirin is safe for children with fever."
         System: REFUTED
         Evidence: 2023 WHO guideline contradicts this claim.
         Confidence: 78%

       How it works:
         Reformulate claim as question → run full pipeline → check if
         retrieved facts support or contradict the original claim.
         Code: enhanced_main.py → verify()

SECTION 6 — WHY EACH ADDITION IN v5 WAS NECESSARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ADDITION 1 — TEMPORAL DECAY [N2]:
  Root cause of v4's failure: score gap between conflicting facts was 0.04.
  Root cause of that: no temporal penalty.
  With decay: gap becomes 1.41 for the aspirin example.
  Conflict resolved instantly. This single addition fixed the core problem.

ADDITION 2 — CORROBORATION [N1]:
  Handles the case where temporal decay alone is insufficient.
  Scenario: 2020 paper (1 source) vs 2019 paper (4 sources agreeing).
  Decay alone: 2020 paper wins (slightly newer).
  With corroboration: 2019 paper with 4 sources might win (more evidence).
  Better reflects real scientific consensus than pure recency.

ADDITION 3 — AUTO SCHEMA [N4]:
  Without it: v4 requires manual domain definition for every new corpus.
  A legal corpus needs different entity types than a medical corpus.
  With auto-schema: drop any documents → system infers its own ontology.
  Made the system genuinely domain-agnostic.

ADDITION 4 — HYBRID RETRIEVAL [N3]:
  Without it: PPR only finds structurally close nodes.
  But what about a query that uses different keywords than the triples use?
  "What drug can't kids take?" might not semantically match "Aspirin CONTRAINDICATED Children"
  BM25 catches keyword overlap; semantic catches meaning.
  RRF ensures both signals contribute.

ADDITION 5 — YEAR-ANCHORED DECAY [N2 FIX]:
  Without this fix: historical queries fail.
  "Was aspirin safe in 1985?" would show 2023 guideline as background.
  2023 guideline contradicts 1985 belief → system would incorrectly say NO.
  With fix: system time-travels to 1985, uses only 1985-era facts.

ADDITION 6 — CONFIDENCE SCORE [N5]:
  v4 gives no calibration.
  Users cannot know when to trust the system.
  3-factor confidence gives a mathematically honest signal.
  Underconfides rather than overconfides → safe for medical/legal use.

ADDITION 7 — EXPLANATION CHAIN [N6]:
  Without it: black box. User doesn't know why a document was removed.
  With it: full audit trail. Every removal has a reason.
  Critical for medical/legal domains where accountability matters.

ADDITION 8 — ADAPTIVE ENTROPY [N9]:
  Without it: always 3×3 = 9 LLM calls for entropy per query (3 paths × 3 samples).
  With it: skip calls for obvious winners/losers.
  Reduces ~35-40% of entropy LLM calls.
  Makes the system ~30% faster without accuracy loss.

ADDITION 9 — CLAIM VERIFICATION [N8]:
  New use case not in v4: fact-checking mode.
  User submits a claim → system says SUPPORTED / REFUTED / UNCERTAIN.
  Directly useful for journalism, legal research, medical second opinions.

SECTION 7 — THE COMPLETE v5 FORMULA REFERENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

COMBINED PATH SCORE:
  score = ref_p × hub_penalty × (1 + ppr_avg) × decay × (1 + log(1+support)×w)

  ref_p       = α × entity_coverage + β × relation_coverage
                where α=0.6, β=0.4
  hub_penalty = (1 - hub_weight) if degree > 10 else 1.0
  ppr_avg     = (ppr[source_node] + ppr[target_node]) / 2
  decay       = exp(-λ × (reference_year - fact_year))
                λ = 0.08 (global, same for all domains)
  corroborate = log(1 + support_count) × corroboration_weight

TEMPORAL DECAY:
  w(t) = e^(-λt)     where t = reference_year - fact_year
  λ = 0.08           half-life = ln(2)/0.08 = 8.66 years

PERSONALISED PAGERANK:
  PPR(v) = (1-d)/|V| + d × Σ PPR(u)/degree(u)   for u in neighbours(v)
  d = 0.85 (damping factor)
  Seeds initialised with PPR = 1/|seeds|
  Run 20 iterations

RECIPROCAL RANK FUSION:
  RRF(d) = 1/(60 + rank_BM25(d)) + 1/(60 + rank_semantic(d))

SHANNON ENTROPY:
  H = -Σ p(x) × log(p(x))          (log base e = nats)
  H_param = entropy of LLM answers without context
  H_aug   = entropy of LLM answers with a specific knowledge path
  ΔH      = H_param - H_aug

CONFLICT SELECTION CRITERION:
  Keep path p if: ΔH(p) > tau
  tau = 0.30 for factual_lookup intent
  tau = 0.35 for safety_check intent
  tau = 0.25 for comparison intent

CONFIDENCE SCORE:
  h_sig = min(ΔH / tau, 2.0) / 2.0
  sup_s = min(log(1 + support) / log(6), 1.0)
  rec_s = exp(-0.05 × (current_year - most_recent_source_year))
  conf  = 0.40 × h_sig + 0.30 × sup_s + 0.30 × rec_s

CORPUS: 6 medical documents, Doc A (1985) says YES, Doc B (2023) says NO.

SECTION 10 — ONE-SENTENCE SUMMARY OF EACH SYSTEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LANGCHAIN RAG:
  Finds text chunks similar to your query and hands them all to the LLM,
  including contradicting ones, with no way to know which is correct.

TRUTHFULRAG v4 (Paper):
  Extracts structured facts into a knowledge graph, retrieves via PageRank,
  and uses entropy to filter useful facts — but cannot resolve conflicts when
  facts score similarly due to missing temporal information.

TRUTHFULRAG v5 (This Project):
  Extends v4 with temporal decay, corroboration counting, hybrid retrieval,
  and a calibrated confidence score so that when documents conflict, the system
  automatically selects the most recent, best-supported fact and explains why
  every other fact was discarded — achieving 90% accuracy vs 56% for LangChain.
