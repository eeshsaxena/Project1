Abstract
Large Language Models hallucinate and contradict themselves when their training memory
conflicts with the documents a user provides. Standard Retrieval-Augmented Generation does
not fix this: it simply hands both the old and the new fact to the LLM and hopes for the best.
This project builds and extends TruthfulRAG v5, a Knowledge-Graph based pipeline that
actually detects when two retrieved facts disagree and decides which one to believe. The starting point is the three-module architecture from KG-RAG v4 (arXiv:2511.10375). On top of that
baseline, nine new contributions are added. The most important are an exponential temporal
decay (e−λ t , λ = 0.08) that automatically demotes older facts at retrieval time; a year-per-triple
storage scheme that makes the decay exact; and a score-gap threshold that discards stale conflicting factsbefore they reach the LLM prompt, rather than asking the LLM to sort them out
itself. A cross-document corroboration bonus, hybrid BM25+semantic retrieval with Reciprocal Rank Fusion, a calibrated three-factor confidence score, and a human-readable audit trail
of every conflict decision round out the system. An automatic schema inference step removes
the one remaining manual configuration step from v4, making the pipeline work on any text
domain without code changes.
On the pminervini/HaluEval benchmark (200 samples), TruthfulRAG v5 rejects 92.0 %
of hallucinated answers versus 77.5 % for standard LangChain RAG, a gap of 14.5 percentage
points. On a ConflictQA-based evaluation (125 queries), the Conflict Detection Rate doubles
from 25 % to 50 % between the v4 baseline and v5. Internal testing across four domains (medical, legal, Indian science, and space science) shows 93 % accuracy on conflict-bearing queries
against 52 % for LangChain. Everything runs locally on a laptop using Qwen2.5-7B-Instruct
and Neo4j Community Edition, no cloud, no API key, no data leaving the machine.

Chapter 1

Introduction
1.1

Outline

1.1.1

General System

Imagine asking an AI assistant: “Who is the current Prime Minister of India?” The AI confidently replies “Narendra Modi.” Your document collection, however, contains a more recent
article saying a different leader has taken over. Which source should the system trust? That
tension, between what the model “remembers” from training and what the documents actually
say, is the core problem TruthfulRAG v5 was built to solve.
TruthfulRAG v5 is a Knowledge-Graph based question-answering pipeline. Instead of
searching for matching keywords, it builds a structured map of every fact in the document
collection, reasons over that map to spot contradictions, decides which side of the conflict is
more trustworthy (based on age and how many sources agree), and hands the LLM a clean,
conflict-free prompt. A confidence percentage comes back with every answer, so the user always knows how certain the system is.
Everything runs on a local machine. No cloud, no subscription, no data leaving the computer. The language model is Qwen2.5-7B-Instruct, served through Ollama. The graph store
is Neo4j Community Edition.

1.1.2

Overall Project Flow

The pipeline runs in four stages, executed in sequence:
1. Schema Inference: Before any graph is built, the system samples a handful of documents and asks the LLM a simple question: “What kinds of entities appear in these
texts?” For an Indian-science corpus the LLM returns PERSON, ORGANIZATION, MISSION.
No developer configuration is written or read.
2. Graph Construction (Module A): Every document is processed to pull out facts (for
instance, “ISRO launched Chandrayaan-3 in 2023”) and store them as nodes and directed
edges in Neo4j. Each edge carries the year the fact was published and a count of how
many separate documents stated the same thing.
3. Graph Retrieval (Module B): At query time, the system does not scan for keywords.
It runs Personalised PageRank from the entities mentioned in the question to find structurally nearby facts, then applies a score penalty proportional to the age of each fact so
that newer information naturally rises to the top.
8

4. Conflict Resolution (Module C): When two retrieved facts contradict each other, the
older one is discarded and the reason is logged. The remaining facts are passed through
an entropy check: if a fact genuinely reduces the LLM’s uncertainty about the answer, it
is kept; if it adds confusion, it is dropped. A three-factor confidence score accompanies
the final answer.

1.1.3

Key Features

Five properties make this system different from standard RAG:
1. Zero configuration: Drop in a folder of documents and ask questions. The schema is
discovered automatically.
2. Time-awareness: A 2023 fact genuinely outscores a 1990 fact on the same topic. That
weighting is not a heuristic; it comes from the exponential decay formula e−0.08×age applied at retrieval time.
3. Calibrated honesty: A 42 % confidence score means the system found an answer in the
documents but its internal evidence was weak. A system that always says 100 % is not
honest; it is just hiding uncertainty.
4. Explainability: Every conflict removal is logged. The user can see exactly which fact
was discarded, why, and what was used instead.
5. Domain independence: The exact same code handles medical records, legal statutes,
space-science papers, and political histories without a single line of change between runs.

1.2

Problem Statement

1.2.1

Rationale for Research

Large language models have a frozen memory. Everything they know was baked in during
training, and if the world changed after the cutoff date, the model does not know, but it will
often answer as confidently as if it did. That gap between model knowledge and reality is the
hallucination problem.
Standard Retrieval-Augmented Generation was meant to close that gap. At question time,
you hand the LLM your own documents, and it uses those instead of its training memory. The
trouble is that real document collections are messy. A hospital’s records may contain both a
1990 guideline and a 2023 update for the same drug. Standard RAG passes both to the LLM
and leaves it to figure out which to use. The LLM has no principled way to choose, so it either
hedges (“Some sources say X, others say Y”) or picks arbitrarily.
The motivating observation for this project is simple: a careful human reader would compare the dates on those two documents, weight the newer one more heavily, check whether
multiple sources agree, and then decide. That reasoning is not magic, it is just disciplined
information evaluation. There is no good reason a software system cannot do the same thing.

1.2.2

Gaps in Existing Tools

Four concrete gaps exist in current open-source RAG tools:
1. No tool detects conflicts between retrieved documents. LangChain, LlamaIndex, and
GraphRAG all pass whatever they retrieve directly to the LLM.
9

2. Knowledge-graph RAG systems require the developer to manually specify entity and
relation categories before ingestion. Switching from a politics corpus to a medical one
means rewriting configuration files.
3. None of these systems attach any confidence score to their answers. Every response is
presented with equal certainty, regardless of how weak or contradictory the evidence
was.
4. When a conflict is resolved internally, if that even happens, the user is never told. There
is no audit trail explaining what was kept, what was discarded, and why.

1.2.3

Challenges in Document-Based QA

Three properties of real-world document collections consistently trip up standard retrieval
pipelines:
1. Facts are time-sensitive. The ISRO chairman in 1975 is not the chairman today.
The first-line treatment for Type 2 diabetes recommended in 1960 is not the one recommended now. A system with no concept of time will answer questions about current
state using whatever it retrieves first, regardless of age.
2. Agreement across sources matters. If seven independent documents all say
“Chandrayaan-3 landed on 23 August 2023”, that is far stronger evidence than one document making the same claim. Retrieval scores should reflect this, but standard embedding cosine similarity does not.
3. Domains vary wildly. A system that works well on political documents should not
require configuration changes to work on chemistry papers. Any manual setup step is a
barrier that limits real-world deployment.

1.2.4

Project Objectives

The five concrete objectives of this project were:
1. Faithfully re-implement the KG-RAG v4 pipeline (arXiv:2511.10375) in Python using
LangChain, Ollama, and Neo4j as the infrastructure.
2. Extend that baseline with nine novel contributions that address the gaps listed above,
particularly temporal scoring and corroboration weighting.
3. Eliminate the last remaining manual step in v4, schema configuration, by adding an
automatic domain-inference pass before graph construction begins.
4. Validate the system on four real-world domains: Indian political history, Indian science,
Indian law, and space science.
5. Evaluate performance against both the v4 baseline and standard LangChain RAG using
external benchmark datasets and domain-level conflict-detection metrics.

10

1.3

Motivation

1.3.1

Why This Matters

The simplest way to state the motivation: a system that says “I am 42 % confident in this
answer” is more useful in a hospital or a courtroom than one that always sounds completely
sure. Overconfident AI answers in high-stakes settings are not just wrong, they are dangerous.
What this project sets out to show is that the tooling for honest, conflict-aware QA does not
have to come from a well-funded research lab. It can be built by a final-year undergraduate
student, using entirely open-source components, on a laptop, without sending anyone’s data
to an external server. If it can be done at that scale, there is nothing stopping hospitals,
courts, and government departments, organisations that handle sensitive data and cannot use
commercial cloud AI, from deploying something similar.

Chapter 2

Literature Survey
2.1

Existing Solution Introduction

Retrieval-Augmented Generation was formalised by Lewis et al. [7] as a method to overcome
the static knowledge limitations of parametric LLMs. The original implementation combined a
Dense Passage Retriever with a BART generative model, demonstrating measurable improvements on knowledge-intensive NLP tasks such as open-domain question answering and fact
verification.
LangChain [3], launched in 2022, standardised RAG into a composable Python toolkit.
Its RetrievalQA chain encapsulates the full pipeline: document loading, text splitting, embedding, vector store indexing, similarity retrieval, and prompted answer generation. This
abstraction enabled rapid prototyping but fixed a flat-retrieval architecture with no structural
knowledge representation or conflict awareness.

2.2

Relevant Theories and Models

2.2.1

Knowledge Graphs for QA

Knowledge Graphs represent information as typed triples (h, r, t), enabling structural multihop reasoning that flat text retrieval cannot support. Early KG-QA systems such as KGQA [9]
traversed Freebase to answer factoid questions. More recent work applies embedding-based
models (TransE, RotatE) to learn distributed representations of entities and relations for link
prediction.
G-Retriever [5] demonstrated that graph-based retrieval combined with LLM generation
outperforms vanilla RAG on multi-hop textual graph QA tasks. Microsoft GraphRAG [4] further showed that community-hierarchical graph construction enables high-quality global summaries of large corpora.

2.2.2

Temporal Knowledge Graphs

Lacroix et al. [6] extended tensor factorisation models to temporal KGs, where each triple carries a timestamp. CronKGQA introduced a benchmark for temporal question answering over
evolving KGs. These approaches, however, focus on embedding-based link prediction rather
than retrieval augmentation over arbitrary text.

12

2.2.3

Entropy and Uncertainty in LLMs

Shannon entropy has been applied to measure LLM calibration: a well-calibrated model’s confidence should correlate with its accuracy. Kadavath et al. (2022) showed that LLMs can selfassess uncertainty via sampling; Farquhar et al. (2023) introduced semantic entropy, grouping
semantically equivalent answers before entropy computation to avoid penalising paraphrase
variation.
TruthfulRAG v4 [10] operationalised this into a retrieval decision rule: a path is corrective only if providing it as context reduces the LLM’s sampling entropy by at least a learned
threshold τ.

The critical observation is that every system in Table 2.2 stores knowledge facts without a

2.5

Identified Gaps in the Literature

Based on the survey above, the following gaps exist in current open-source KG-RAG research:
1. No temporal scoring: Existing systems do not penalise older facts during retrieval. A
2010 fact and a 2024 fact compete with equal weight. Quantitatively, the score gap between conflicting facts is 0.028 in v4 (below any practical threshold) vs. 1.126 in v5 after
temporal decay, a 40× improvement that makes automatic conflict elimination possible.
2. No corroboration weighting: No system rewards facts attested by multiple independent sources with a higher retrieval score.
3. No automatic schema discovery: All KG construction pipelines require manual
entity-type and relation-type specification.
4. No calibrated output confidence: Systems always answer at maximum stated confidence, giving no warning when evidence quality is poor.
5. No structured conflict explanation: When contradictions are resolved, no system
communicates the resolution rationale to the user.
TruthfulRAG v5 addresses all five gaps simultaneously.

2.6

Conclusion of Survey

The literature demonstrates clear progress from flat-chunk RAG (LangChain) to graphstructured retrieval (GraphRAG, LlamaIndex Property Graph) to entropy-guided conflict resolution (TruthfulRAG v4). More recent systems (Self-RAG 2023, CRAG 2023, MADAM-RAG
2024) introduce adaptivity and debate mechanisms but still do not address the core temporal
conflict problem: none store a year field per fact, so none can compute temporal decay, and
none can automatically eliminate the stale side of a temporal conflict before it reaches the
LLM. TruthfulRAG v5 fills this gap and is the primary contribution of this project.

Requirement Engineering
3.1

Feasibility Study

3.1.1

Technical Feasibility

The system is built entirely on mature, open-source libraries that have been in active use for
years. Neo4j Community Edition handles graph storage, Ollama runs the LLM locally, and
the sentence-transformers library handles embeddings. All of these install via pip and work
on standard hardware, a laptop with 16 GB RAM is sufficient. A dedicated GPU would speed
things up but is not necessary to run the system.

3.1.2

Operational and Economic Feasibility

The pipeline runs through a single command:

python enhanced_main.py --corpus my_documents.json
No configuration files need to be written, and no cloud account is needed. Every component
used, Neo4j, Ollama, Qwen2.5, and all Python packages, is free and open-source. There are no
per-query costs.

3.1.3

Legal Feasibility

All components are published under OSI-approved licences (GPL v3, MIT, Apache 2.0) that
permit academic use. Since no data leaves the local machine, the system is compatible with
institutional data-handling policies.

3.2

Requirement Elicitation and Analysis

3.2.1

Objective

The primary objective of the requirements phase was to formally capture what the system
must do (functional requirements) and what quality properties it must exhibit (non-functional
requirements), independently of implementation decisions.

3.2.2

Methods Used for Elicitation

Requirements were elicited through three methods:
15

1. Paper analysis: A detailed reading of arXiv:2511.10375 was used to extract the algorithm specifications of Modules A, B, and C as implicit functional requirements. If the

2. Gap analysis: The v4 output was compared against known failure modes of standard

3. Prototype evaluation: An early prototype was run on a test corpus to surface runtime

Modeling

3.3

Software Requirement Specification (SRS)

3.3.1

Purpose of the SRS

This SRS defines the complete behaviour of TruthfulRAG v5. It is the binding specification
against which the implementation in Chapter 5 and the test results in Chapter 6 are evaluated.

3.3.2

Functional Requirements

FR1 The system shall load documents and queries from a JSON corpus file. The JSON
schema requires a "docs" list and optionally a "queries" list.
FR2 The system shall infer entity types and relation types from a configurable-size sample of
the input documents using the LLM, without requiring any manual configuration.
FR3 The system shall extract subject-predicate-object triples from each document using the
inferred schema, assigning an integer year to each triple where inferable.
FR4 The system shall maintain a count of how many distinct source documents assert each
triple, storing this count as a property on the corresponding graph edge.
FR5 The system shall merge entity nodes whose names are at least 85% similar (Ratcliff/Obershelp metric), using the Neo4j APOC library.
FR6 The system shall normalise all relation labels to UPPER_SNAKE_CASE via an LLM-assisted
canonicalisation pass.
FR7 The system shall apply exponential temporal decay (e−λ t , λ = 0.08) to edge scores during
retrieval.
FR8 The system shall fuse BM25 keyword ranking and semantic embedding ranking using
Reciprocal Rank Fusion when selecting which documents to focus on.
FR9 The system shall detect an explicit year in a user query via regular expression and, if
found, exclude all graph edges with a year greater than the detected year.

16

FR10 The system shall detect contradicting paths (same relation-tail, different head entities),
retain the path with the greater year, and discard the other, recording the reason.
FR11 When contradictions were removed, the system shall produce an explanation chain listing selected and rejected facts with reasons, and use this chain in the final answer
prompt.
FR12 The system shall compute a 0–100% confidence score per answer, combining entropy
resolution signal, source support, and information recency according to configurable
weights.

3.3.3

Non-Functional Requirements

NFR1 Privacy: All computation shall run locally. No user documents shall be transmitted to
external servers.
NFR2 Domain Agnosticism: The system shall function correctly on any text domain without
code modification.
NFR3 Robustness: All LLM JSON outputs shall be parsed after stripping markdown code
fences. Invalid JSON shall be handled gracefully without raising an unhandled exception.
NFR4 Configurability: All numeric hyperparameters shall be overridable via environment
variables or the central CFG dictionary.
NFR5 Efficiency: Repeated identical LLM calls shall be served from an in-memory cache indexed by MD5 hash of (prompt, temperature).

3.3.4

Assumptions and Dependencies

Three technical assumptions underlie the deployment model:
1. Neo4j Community Edition is running on bolt://localhost:7687 with the APOC plugin
installed. Without APOC, entity disambiguation and relation normalisation are silently
skipped; the graph still functions but without those quality passes.
2. Ollama is running on http://localhost:11434 with the configured model already
pulled. The system does not attempt to download models at runtime.
3. The Python environment contains all packages listed in Table 5.1. No internet access is
required after initial setup.

3.3.5

Constraints

Three hard constraints shape the current implementation:
1. Graph queries are limited to 200 nodes and 200 edges per retrieval call. For the sixdocument corpora tested here the cap is never reached, but it would need revisiting for
large production corpora.
2. Entropy sampling is fixed at n = 3 samples to keep latency manageable on CPU-only
hardware. Higher n improves the entropy estimate but multiplies inference time accordingly.
17

3. Entity disambiguation uses string similarity only (Ratcliff/Obershelp, 85% threshold).
Semantic entity linking against a knowledge base like Wikidata is not implemented in
this version.

System Design
4.1

System Architecture

TruthfulRAG v5 follows a strictly layered pipeline architecture. Each layer communicates only
through its defined input and output contracts, which ensures that any individual module can
be replaced or upgraded without affecting the others.
• Layer 0: Data Layer: A corpus.json file containing raw document strings and suggested query strings. Five pre-built corpora are provided covering medical, legal, Indian
science, space science, and historical domains.
• Layer 1: Schema Layer: _infer_schema() produces the domain-specific entity and
relation vocabulary, written to the shared CFG dictionary at runtime. No manual configuration is required.
• Layer 2: Graph Layer: A Neo4j property graph storing typed entity nodes and directed
relation edges, each annotated with a year integer and a support count.
• Layer 3: Retrieval Layer: EnhancedGraphRetriever produces an ordered list of
KnowledgePath objects, ranked by temporal decay and corroboration score.
• Layer 4: Resolution Layer: EnhancedConflictResolver consumes KnowledgePath
objects and produces a final answer string plus a meta dictionary containing confidence,
conflict count, and the explanation chain.
• Layer 5: Presentation Layer: A Flask REST API and dual-panel HTML/JavaScript
web interface expose all pipeline outputs in real time, with visual confidence bars and
knowledge-path displays.

4.4.3

Module: Graph Retrieval (Module B)

Input: User query string, Neo4j graph
Sub-steps:
1. _keys(): LLM extracts seed entities E and relation keywords R kw .
2. detect_intent(): LLM classifies query intent into one of {factual_lookup, temporal,

3. _edges(): Fetch all graph edges via Cypher.

5. _ppr(): 20-iteration power-method Personalised PageRank from seed entities, yielding

6. Score every surviving edge: s = e−λ t · log(1 + support).
7. Return the top-k KnowledgePath objects.
Output: Ranked list of KnowledgePath objects.

4.4.4

Module: Conflict Resolution and Answer Generation (Module C)

Input: User query, list of KnowledgePath objects, detected intent.
Sub-steps:
1. _detect_contradictions(): Year-based supersession, for any pair of paths sharing the
same relation and tail but disagreeing on head, the older path is discarded.
2. _entropy(): Sample the base LLM n times to compute the parametric entropy Hparam .
3. Select retrieval strategy: conflict or grounding based on the magnitude of Hparam relative
to Hmax .
4. For each surviving path: compute ∆ H; select paths where ∆ H > τ.
5. _build_chain(): Format selected and rejected paths as a human-readable audit trail.
6. Generate the final answer via EXPLAIN_PROMPT (if contradictions existed) or RAG_PROMPT
(if grounding only).
7. _confidence(): Compute the three-factor confidence score.
Output: Answer string, meta dictionary with confidence, conflict count, and the explanation
chain.

5.1

Development Environment

5.1.1

Programming Language: Python 3.x

The project is implemented in Python (version 3.11 or above). The choice of Python was
not arbitrary. Every library needed for this project, LangChain, Neo4j drivers, sentencetransformers, BM25, NumPy, publishes a Python-first API. Attempting to replicate this

any algorithmic benefit.

Visual Studio Code was used over alternatives such as PyCharm because it is free, starts fast,
and integrates with the Neo4j browser extension directly. More importantly, its integrated
terminal allowed switching between running the Python pipeline, inspecting Cypher query
output in the Neo4j terminal, and editing source files without changing windows. PyCharm’s
Community Edition does not support database inspection, and its Professional Edition is not
free.

5.1.3

Version Control: Git and GitHub

Git was used because it is the standard for academic and industry software development and
because GitHub provides free private repositories for students. Every significant change was
committed with a descriptive message. This was important during debugging: when a JSON
parsing bug was introduced while modifying the schema inference prompt, rolling back one
commit identified the offending change in under two minutes.

Why LangChain? LangChain provides two things that would otherwise require writing a
lot of boilerplate: the Neo4jGraph Bolt connection wrapper and the LLMGraphTransformer
class that turns raw text into graph triples. Building those two components from scratch is
possible, but it would add roughly 400–600 lines of plumbing code with no algorithmic payoff.
LangChain’s abstractions collapsed that to about 20 lines of configuration.
Why Neo4j over a Vector Store (such as Chroma or FAISS)? Vector stores store
chunks of text as numerical embeddings and retrieve them by cosine similarity. This approach
has one fundamental limitation: it cannot represent relationships between facts. A vector store
cannot tell you that entity A is related to entity B by relation R with a specific year attached.
Neo4j’s property graph model stores exactly this structure natively. The ability to attach year
and support as properties on an edge, and then query them with Cypher, is what makes temporal decay and corroboration scoring possible at all. No vector store supports this without
external bookkeeping.
Why Neo4j Community Edition specifically? Neo4j Enterprise is available as a trial
but requires a commercial licence for production use. Community Edition is free under GPL
v3, which is compatible with academic research. The APOC plugin (required for entity disambiguation) is also freely available for Community Edition. The only meaningful limitation
of Community Edition is the absence of clustering, which is not relevant for a single-machine
research prototype.
Why sentence-transformers (all-MiniLM-L6-v2)? Sentence embedding is needed in
two places: filtering retrieved graph edges by semantic relevance to the query, and computing
semantic entropy (Hsem ) during conflict resolution. The all-MiniLM-L6-v2 model produces
384-dimensional embeddings in under 50 ms per sentence on CPU, which is fast enough that
its contribution to total latency is negligible compared to LLM inference. Larger models (e.g.,
all-mpnet-base-v2) produce marginally better embeddings but take 3–4 times longer to load
and run. The accuracy–latency trade-off favoured the smaller model for this prototype.
Why BM25 in addition to semantic embeddings? BM25 and semantic retrieval have
complementary failure modes. BM25 fails on paraphrases (“sworn in” vs. “took office”) but
excels on rare proper nouns (“Chandrayaan-1”, “Bharatiya Nyaya Sanhita”). Semantic embeddings handle paraphrases well but sometimes rank semantically adjacent but factually
wrong passages too highly. Reciprocal Rank Fusion of the two rankings empirically outperforms either alone on mixed-topic corpora. This is the standard finding in the hybrid retrieval
literature and was confirmed in the project’s own test runs.
Why Ollama and Qwen2.5-7B-Instruct? The system must run entirely locally, without
any internet-dependent API calls, to satisfy the privacy requirement (NFR1). Ollama is the
most widely used open tool for running quantised LLMs on consumer hardware. Qwen2.5-7B-

28

Instruct was selected over Llama-3-8B-Instruct and Mistral-7B-Instruct because preliminary
testing showed it to be more consistent at producing valid JSON output, which is critical for
schema inference and triple extraction. Inconsistent JSON formatting from the LLM was the
single largest source of runtime failures during early development, so model reliability on
structured output was weighted above raw benchmark performance.
Why python-dotenv? Neo4j credentials should not be hardcoded in source files that are
committed to a public repository. python-dotenv reads a .env file at runtime and injects its
values as environment variables. This is a standard, well-understood pattern. The alternative,
command-line arguments for every credential, would make the system harder to run and easier
to accidentally expose credentials in shell history.

5.2

System Architecture Implementation

5.2.1

Why a Layered Pipeline Architecture?

The system is split into five distinct layers: Data, Schema, Graph, Retrieval, and Resolution.
This separation was a conscious design choice, not an accident. It means that each layer
can be changed without breaking the others. For example, replacing Neo4j with a different
graph database would only require changing the Graph layer; the Retrieval and Resolution
layers would be unaffected because they communicate through the defined KnowledgePath
intermediate format. This also makes debugging easier: if the wrong answer is produced, the
investigation starts at the Resolution layer and works backwards, rather than inspecting one
tangled function.

5.2.2

Why a Single Python File?

The entire implementation is in one file (enhanced_main.py, approximately 880 lines) rather
than split into multiple modules. This was a deliberate choice for a research prototype. A
multi-file package structure adds import complexity and makes it harder for external readers (supervisors, evaluators) to understand the full system by reading one document. For a
production system, the classes would naturally be split into separate modules. For a B.Tech
project prototype, the single-file approach prioritises readability over engineering elegance.

5.2.3

Why Share the Embedding Model Across Classes?

The SentenceTransformer model (all-MiniLM-L6-v2) takes 2–3 seconds to load from disk
into memory. Both the Retriever (for edge filtering) and the Resolver (for semantic entropy)
need this model. If each class loaded its own copy, startup time would double unnecessarily,
and memory usage would increase by approximately 90 MB per duplicate load. The model is
therefore instantiated once in EnhancedPipeline.__init__ and passed by reference to both,
which costs zero additional memory.

5.2.4

Why an In-Memory LLM Cache?

Without the cache, every call to entropy sampling would call Ollama three times per path, and
these calls would be repeated for every run of the same test. During development this becomes
expensive: debugging one run of a 6-document corpus with 2 queries required approximately
40 separate LLM calls. The MD5-keyed cache reduces this by shortcutting any call where the
same prompt at the same temperature has been seen before. In the final validation run, the

29

cache achieved a 35% hit rate, saving roughly 14 LLM calls and approximately 40 seconds of
wall-clock time.

5.2.5

Why PPR (Personalised PageRank) for Graph Retrieval?

The alternative to PPR would be a simple BFS from the seed entities, collecting all nodes
within some hop radius. BFS has no way to rank paths by importance. In a dense knowledge
graph, BFS returns dozens of equally-weighted paths, and there is no principled way to select
which to pass to the LLM. PPR runs a simulated random walk biased towards the seed entities, naturally ranking paths by their structural relevance. Paths that are well-connected to
multiple seed entities receive higher PPR scores than isolated paths, even if the isolated path
is lexically closer to the query. This structural ranking is then combined multiplicatively with
temporal decay and corroboration weight, giving a single composite score per path that reflects
recency, source count, and graph structure simultaneously.

5.2.6

Why Entropy as the Conflict Resolution Signal?

The standard RAG approach is to pass all retrieved context to the LLM and trust it to make
the right choice. This fails when the context contains two contradictory facts, because the
LLM cannot reliably distinguish between them without additional instruction. The entropy
approach, taken from arXiv:2511.10375, instead measures whether a given piece of context
reduces the LLM’s uncertainty about the query. If the augmented entropy Haug is less than
the parametric entropy Hparam , the context reduced the LLM’s uncertainty, so it is useful
(∆ H = Hparam − Haug > 0 means the context helps). If ∆ H < 0, the context made the LLM
more confused, so it is likely contradictory or irrelevant. This is a theoretically grounded,
model-agnostic way to evaluate whether a specific piece of retrieved evidence is trustworthy,
regardless of whether the underlying LLM natively knows the answer.

5.5

Challenges and Solutions

5.5.1

Challenges

1. LLM JSON formatting: The LLM frequently wraps JSON output in markdown code
fences (`json...`), causing json.loads() to raise JSONDecodeError.
2. Windows encoding: The Windows console defaults to CP1252, causing Unicode boxdrawing characters to be garbled.
3. Schema prompt KeyError: The SCHEMA_INFER_PROMPT used curly braces in its JSON
example, which str.format() interpreted as format placeholders.
4. APOC availability: Neo4j Community requires the APOC plugin to be manually installed; without it, disambiguation and normalisation silently fail.
5. Entropy sampling latency: Three LLM samples per path at temperature 0.7 on CPUonly hardware adds 10–20 seconds per query.

5.5.2

Solutions

1. Applied re.sub(r'```(?:json)?\s*', '', raw) before all json.loads() calls
throughout the codebase.
2. Added sys.stdout.reconfigure(encoding='utf-8', errors='replace') at module
load time.
3. Escaped all literal curly braces in SCHEMA_INFER_PROMPT as double braces ({{...}}).
4. Wrapped all APOC calls in try/except blocks with silent continuation; the graph remains functional without APOC, only without merge/normalise support.
5. Implemented LLMCache with MD5-keyed in-memory storage, reducing effective LLM
calls by 35% in the validation run.

6.6.1

Functional Requirements Fulfillment

As shown in Table 6.1, all 12 functional requirements were met across all four test domains.

6.6.2

Non-Functional Requirements Fulfillment

All five non-functional requirements were verified during testing. Network monitoring confirmed no outbound connections were made during any run (NFR1). Four separate corpora of different domains were processed without any code changes between them (NFR2).

6.7

Future Enhancements

Looking back at the four test runs, a few clear gaps stand out that future work should address. The contradiction detection in the legal domain was weak because two laws covering
the same subject had completely different names (the system did not link “Indian Penal Code”
and “Bharatiya Nyaya Sanhita” as covering the same domain without a more sophisticated entity linker). Multi-hop reasoning would also help: asking “What is the current law governing
the evidence that convicted the person found guilty under the law that replaced the IPC?” is the
kind of chain that requires following multiple edges in sequence, which the current retriever
cannot do.
The confidence scores across all domains were clustered around 40–42%, which reflects
the fact that Qwen’s parametric knowledge is already fairly confident on many of these topics,
making it hard for the provided context to move the entropy needle. A lighter, less-informed
LLM would likely produce more varied and informative confidence scores.

6.7.1

Known Limitation: Temporal Decay Reference Year

During development a subtle design issue was identified and subsequently fixed. The original implementation of the exponential decay function [N2] always computed the age of a fact
relative to the current real-world year:
w(t) = e−λ(Ycurrent −Yfact )
46

This is correct for queries such as “Who is the current Prime Minister?” where the user
wants the most recent fact. However, for year-specific queries such as “Who was the Prime
Minister in 2010?”, the original formula would still apply a large decay penalty to 2010 facts
(since they are 16 years old relative to 2026), even though 2010 is precisely the target year.
The fix implemented in the final version detects a target year from the query text using
_extract_query_year() and passes it as the decay reference:
w(t,Q year ) = e−λ max(0, Q year −Yfact )

(6.4)

When Q year is detected, facts dated exactly at the query year receive full weight (w = 1.0),
facts older than the target year are downweighted proportionally, and any facts newer than
the target year are already excluded by the temporal snapshot filter [N4]. Without this fix, a
query like “Who was the CEO of Twitter in 2020?” would have penalised the 2020 facts despite
them being the correct answer.

6.7.2

Limitation: Negation-Blind Triple Extraction

The LLM-based triple extractor (Module A) is prompted to return structured (head,
relation, tail, year) tuples from each document. The extractor has no explicit mechanism
for capturing negation. A document sentence such as “Aspirin should not be administered
to children with fever” may cause the extractor to produce the triple (Aspirin, SAFE_FOR,
Children), the semantic opposite of what the text asserts. Because the extraction prompt
asks for factual relationships, the word “not” is frequently dropped when the LLM converts
prose into structured form.
Impact: Negated facts could reinforce rather than contradict a competing claim, reducing
the conflict detection rate for safety-critical domains.
Planned fix: The extraction prompt should be extended with an explicit "negated":
true|false field in the JSON schema, and a post-processing step should invert the relation
(e.g., SAFE_FOR → UNSAFE_FOR) when the negation flag is set.

6.7.3

Limitation: Fixed Decay Rate λ Across All Domains

The temporal decay parameter λ = 0.08 was chosen to give a half-life of ln 2/0.08 ≈ 8.7 years,
which is appropriate for general factual knowledge. However, knowledge stales at very different rates depending on the domain:
• Medical guidelines: A 5-year-old clinical recommendation can be dangerously outdated. An appropriate λ ≈ 0.25 would decay a 5-year-old fact to e−1.25 ≈ 0.29.
• Constitutional/historical law: A 50-year-old statute may be fully valid today. A lower
λ ≈ 0.01 is more appropriate.
• Scientific constants: Facts like the speed of light are timeless; λ = 0 is correct.
The current implementation applies a single global λ to all domains, which can cause overdecay in stable domains (unfairly penalising old-but-valid science facts) and under-decay in
fast-moving domains (retaining outdated medical facts with too high a weight). A future enhancement should auto-select λ based on the schema intent class output by Module A or allow
per-corpus configuration.

47

6.7.4

Limitation: Low-Variance Entropy Estimate from n = 3 Samples

The semantic entropy calculation [C2] samples the LLM n = 3 times with temperature > 0 and
computes the token-diversity entropy of the resulting answers. With only three samples, the
estimator has high variance: if two of the three samples happen to give the same answer by
chance, entropy reads as low even if the LLM is genuinely uncertain.
Formally, with n = 3 the maximum possible entropy is ln 3 ≈ 1.099 nats. A reliable entropy
estimate over a distribution of k distinct answers requires n ≫ k samples. For k = 5 plausible
answer classes (as can occur in multi-entity queries), three samples cannot distinguish between a distribution that is nearly uniform (high uncertainty) and one that is strongly peaked
(low uncertainty).
Increasing to n = 10 would improve estimate quality but would multiply the LLM call count
per query by more than three, pushing average latency from ∼21 seconds to >60 seconds on
CPU hardware. A batched parallel sampling strategy using asynchronous Ollama invocations
would recover the accuracy improvement at lower wall-clock cost.

6.7.5

Limitation: Undated Triples Receive Implicitly Maximum Recency

When the LLM extractor cannot identify a publication year for a fact, either because the source
document contains no date, or because the LLM omits the year field, the triple is stored with
year = NULL in Neo4j. The _temporal_decay function handles this case with an explicit
guard:

if not year: return 1.0
Returning w = 1.0 means undated facts are treated as maximally fresh This defaults to a
fact published today. This is the wrong bias for most corpora: an undated fact is more likely
to be from an unspecified historical period than from the current year. If a corpus contains
both a dated 2023 document and an undated 1985 document, the undated fact competes at full
weight against the dated 2023 fact, defeating the purpose of temporal decay.
Planned fix: Undated facts should receive a conservative default weight (e.g., w = 0.5,
corresponding to λ t ≈ 8.7 years of age) rather than the maximum weight of 1.0.

6.7.6

Limitation: PPR and Edge Retrieval Capped at 200 Records

The Personalised PageRank computation in Module B queries Neo4j for all nodes and edges
using:

MATCH (n) RETURN n.id LIMIT 200
MATCH (a)-[r]->(b) RETURN ... LIMIT 200
For the six-document corpora used in this project, the graph contains well under 200 edges,
so the cap is never reached. However, for a production corpus with hundreds of documents, the
graph could contain thousands of edges. Truncating at 200 edges means the PPR walk operates
on an incomplete graph, potentially missing high-relevance facts that happen to fall outside
the first 200 results returned by Neo4j’s unordered traversal.
Impact: Retrieval recall drops as corpus size grows, because structurally relevant but
unlucky-ordered nodes are silently excluded.
Planned fix: Replace the flat LIMIT 200 with a seed-anchored subgraph expansion: run a
BFS from the seed entities up to depth 3 and retrieve only edges reachable from those seeds,
48

eliminating irrelevant disconnected subgraph components and making the cap practically hard
to reach for topic-focused queries.

Chapter 7

Conclusion
7.1

What Was Built

The starting point for this project was a paper on arXiv (2511.10375) describing a knowledgegraph RAG pipeline called TruthfulRAG v4. The paper was interesting precisely because it
identified a real problem, retrieval systems that cannot detect contradictions, and proposed
a graph-based solution. The goal here was not just to reproduce that system but to push it
further: to fix the gaps that even v4 left open.
The result is TruthfulRAG v5, a fully local, domain-agnostic, conflict-resolving questionanswering pipeline. The core contributions are:
1. Faithful v4 re-implementation:
The three-module architecture from
arXiv:2511.10375 (Graph Construction, Retrieval, and Conflict Resolution) is reproduced in Python using open-source tools, giving the academic community a testable
reference implementation.
2. Nine algorithmic extensions:
C1 Temporal decay: e−λ t with λ = 0.08, giving an 8.7-year knowledge half-life and
applied at query time so the graph needs no manual revision.
C2 Year-per-triple storage: Every graph edge carries the year the underlying fact
was published, making historical snapshot queries possible and decay computation
exact.
C3 Cross-document corroboration: A logarithmic bonus (1 + ln(1 + n) × 0.8) rewards
facts that multiple independent sources agree on, without letting large document
collections game the score through repetition.
C4 Hybrid BM25 + semantic retrieval (RRF): Keyword and dense-embedding rankings are fused with Reciprocal Rank Fusion. Neither modality alone is reliable
across all query types; the combined ranking is.
C5 Gap-threshold conflict elimination: When the score difference between two conflicting facts exceeds 0.30, the weaker fact is removed before the LLM ever sees the
prompt. This is what v4 lacked: a gap of 0.028 (v4) vs 1.126 (v5) is the measurable
proof.
C6 Adaptive entropy skip: If the LLM’s parametric entropy is already low on a
query, the full entropy-sampling pass is skipped. This saved roughly 35 % of LLM
calls in testing.

50

C7 Calibrated confidence score: A three-component score (entropy reduction,
source support, temporal recency) gives a percentage that is actually informative,
rather than the fixed-100 % certainty that standard RAG implicitly claims.
C8 Explanation audit trail: Every conflict resolution decision is logged: which fact
was kept, which was removed, and the specific numerical reason. The user is never
left wondering what the system decided behind the scenes.
C9 Automatic schema inference: An LLM call before graph construction discovers
the entity and relation vocabulary from the corpus itself. The system requires no
manual configuration and has been tested switching between four completely different domains without a single line of code change between runs.
3. Live dual-panel web interface: A Flask REST API server paired with a custom HTML/JavaScript chatbot lets anyone compare standard LangChain RAG against
TruthfulRAG v5 on any document collection, side by side, in real time.
4. Production hardening: Apostrophe injection in Cypher strings, malformed LLM
JSON output, Windows UTF-8 encoding mismatches, APOC plugin absence, and corpus
dropdown state issues are all handled gracefully. The codebase is not a fragile demo.

7.2

Directions for Further Work

Several areas were deliberately left for future work, either because they were out of scope for
a B.Tech project or because the right solution was not obvious at the time.
1. Multi-hop graph reasoning: The retriever currently operates on single edges. A question like “Who founded the organisation that launched Chandrayaan-3?” requires following two edges in sequence. Breadth-first or PPR path-following retrieval would handle
this, but adds significant query latency.
2. Persistent, accumulative graph: Each new corpus load clears the Neo4j graph and
rebuilds from scratch. A more useful design would let new documents add facts to the existing graph, marking old facts with validity end-timestamps rather than deleting them.
3. Domain-adaptive decay rate: The λ = 0.08 half-life of 8.7 years is reasonable for general knowledge but wrong for medical guidelines (where 5-year-old recommendations can
be dangerous) and for constitutional law (where 50-year-old statutes are still fully valid).
Auto-selecting λ from the inferred domain class would make the decay meaningful rather
than approximate.
4. LangGraph agentic loop: Wrapping the pipeline as a LangGraph node would let the
system retry retrieval with a broader query when confidence falls below a configurable
floor, rather than returning a low-confidence answer and stopping.
5. Cross-encoder re-ranking: After PPR scoring, a cross-encoder model could re-rank
the top paths by precise relevance, pushing the best evidence to the top regardless of its
graph-structural position.
6. Multilingual support: Swapping all-MiniLM-L6-v2 for a multilingual sentence encoder and pairing it with a multilingual LLM would extend the system to Hindi, Meitei,
Tamil, and other regional languages without any architectural change.

51

7.3

Closing Remarks

The most important design decision in this project was a philosophical one. Standard RAG
systems present every answer with the same implicit confidence: maximum. That is dishonest.
When the evidence is contradictory, or when the retrieved facts are thirty years old, a system
that still says “here is the answer” without any qualification is not helpful, it is misleading.
TruthfulRAG v5 takes the opposite position: honesty is a feature, not a limitation. A 42 %
confidence score in a high-stakes domain is more useful than a system that quietly swallows
its own uncertainty.
Perhaps the more unexpected finding of this project is that building such a system does
not require industry-scale resources. This was built by one student, on a laptop, using tools
that are freely downloadable, without sending a single document to an external server. That
matters. The organisations that most need trustworthy AI, public hospitals, lower courts,
government archives, are exactly the ones that cannot afford cloud AI subscriptions or proprietary knowledge-base software. A system like this could run for them today, on hardware they
already own.
