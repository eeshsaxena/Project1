\section{System Architecture}

TruthfulRAG v5 is built as a six-layer pipeline in which each layer has a
strictly defined role. A document corpus enters at Layer~0 and an audited,
confidence-annotated answer exits at Layer~5. Because adjacent layers
communicate only through well-defined data structures, any single layer can
be swapped or upgraded without touching the rest of the system.

The six layers and their responsibilities are described below and illustrated
in Figure~\ref{fig:arch}.

\begin{itemize}
    \item \textbf{Layer 0 --- Data.} The entry point is a \texttt{corpus.json} file
    that bundles raw document strings together with an optional list of suggested
    queries. Five pre-built corpora are provided (medical, legal, Indian science,
    space science, and political history), but any plain-text collection in the
    same format will work without code changes.

    \item \textbf{Layer 1 --- Schema.} Before a single document is processed, the
    \texttt{\_infer\_schema()} function samples a handful of documents and asks the
    LLM what entity types and relation types appear in the text. The answer
    is written into the shared \texttt{CFG} dictionary and drives all subsequent
    extraction. This is what makes the pipeline domain-agnostic: switching from a
    medical corpus to a legal one requires no code changes at all.

    \item \textbf{Layer 2 --- Graph.} Every document is processed to extract
    subject-predicate-object triples. Each triple is stored as a directed edge in
    Neo4j, carrying the publication year of the source fact and a count of how many
    independent documents assert the same claim.

    \item \textbf{Layer 3 --- Retrieval.} At query time, Personalised PageRank seeds
    from the query's key entities and walks the graph to find structurally relevant
    facts. A cosine-similarity filter removes edges unrelated to the query's relation
    keywords, and an exponential temporal decay formula down-weights older facts.

    \item \textbf{Layer 4 --- Resolution.} The retrieved facts are checked for
    contradictions: if two facts share the same relation and object but differ on
    the subject, the one with the older year is discarded. Entropy sampling then
    verifies that the remaining facts actually reduce the LLM's uncertainty before
    they are included in the final prompt.

    \item \textbf{Layer 5 --- Presentation.} A Flask REST API serves the pipeline
    over HTTP at \texttt{http://localhost}. A web interface allows a user to
    submit queries and view the answer together with the confidence score,
    conflict count, and the audit trail of facts used or removed.
\end{itemize}

\subsection{Knowledge Graph Data Model}

Anyone familiar with relational databases will recognise the Entity-Relationship
(ER) diagram --- the standard tool for sketching out tables, columns, and
foreign-key links before writing a single line of SQL. Neo4j is not a relational
database; its native equivalent is the \textbf{Property Graph Model} diagram,
which describes node labels, relationship types, and the properties each can carry.

The schema chosen for TruthfulRAG v5 is deliberately small: one node type and
one relationship type. That simplicity is intentional --- short Cypher queries,
no join complexity --- but the five properties on every edge do considerable work
behind the scenes, carrying everything the scoring and conflict-resolution logic
needs to make a decision.

One detail worth noting: \texttt{weight} is never written to the database.
It is recalculated fresh at every query using $w = e^{-0.08 \times \text{age}}$,
where \emph{age} is the number of years elapsed since the stored \texttt{year}.
The \texttt{support} counter works the other way --- it accumulates
permanently, incrementing each time a new source document confirms the same
fact through the Cypher \texttt{MERGE} operation.

dependencies. Figure~\ref{fig:deploy} shows the three co-located

cloud-based RAG systems.

\section{Data Flow Diagrams (DFD)}

\subsection{Level 0 Data Flow Diagram (Context Diagram)}

At the highest level, TruthfulRAG v5 has three external entities:

\begin{itemize}
    \item \textbf{User}: Provides a corpus file and receives the final answer
    with confidence score.
    \item \textbf{Ollama (LLM Runtime)}: An external process running locally,
    providing LLM inference.
    \item \textbf{Neo4j (Graph Database)}: An external process running locally,
    providing graph storage and Cypher query services.
\end{itemize}

Data flows at Level 0:
\begin{enumerate}
    \item User $\rightarrow$ System: \texttt{corpus.json} (documents + queries)
    \item System $\rightarrow$ Ollama: Prompts for schema inference, triple extraction,
    entropy sampling, and answer generation
    \item Ollama $\rightarrow$ System: LLM text responses
    \item System $\rightarrow$ Neo4j: Cypher write queries (\texttt{MERGE} node/edge)
    \item Neo4j $\rightarrow$ System: Cypher read results (paths, node identifiers)
    \item System $\rightarrow$ User: Answer text, confidence percentage, conflict count,
    and knowledge-path audit trail
\end{enumerate}

\section{Module Design}

\subsection{Module: Pre-Pipeline Schema Inference [A4+]}

\textbf{Input}: List of document strings, \texttt{cfg[``schema\_infer\_sample'']} (default 5)\\
\textbf{Process}: Sample the first $n$ documents, concatenate with separators, call the LLM
with \texttt{SCHEMA\_INFER\_PROMPT}, parse the JSON response, strip markdown fences,
and write the inferred entity types and relation types to \texttt{cfg}.\\
\textbf{Output}: Updated \texttt{cfg[``schema\_entity\_types'']} and
\texttt{cfg[``schema\_relation\_types'']}\\[0.3cm]
\textbf{Why this module exists}: Without schema inference, the extraction prompts
contain generic placeholder type names that produce inconsistent triples.
Inferred schema names (e.g., ``Pathogen'', ``causes'') produce tightly typed graphs
that PPR can traverse more accurately.

\subsection{Module: EnhancedGraphConstructor}

\textbf{Input}: Document list, CFG, Neo4j connection, LLM\\
\textbf{Process}:
\begin{enumerate}
  \item Call \texttt{LLMGraphTransformer} (LangChain) to extract a first pass of triples.
  \item For each document, call \texttt{TRIPLE\_EXTRACT\_PROMPT} directly and accumulate
        typed, year-annotated \texttt{TripleRecord} objects.
  \item Write all records to Neo4j with \texttt{MERGE} (idempotent).
  \item Run entity disambiguation via APOC similarity merge.
  \item Normalise relation labels to canonical forms via a second LLM call.
\end{enumerate}
\textbf{Output}: A populated Neo4j knowledge graph.

\subsection{Module: HybridRetriever [N3]}

\textbf{Input}: Query string, Neo4j graph, embedding model, BM25 index\\
\textbf{Process}: Rank graph edges by hybrid BM25 + cosine similarity score;
return top-$k$ edges as candidate triples.\\
\textbf{Output}: Filtered edge list for PPR seeding.

\subsection{Module: EnhancedGraphRetriever}

\textbf{Input}: Query string, candidate edge list, CFG\\
\textbf{Process}:
\begin{enumerate}
  \item Extract seed entities and relation keywords via \texttt{\_keys()}.
  \item Classify query intent via \texttt{detect\_intent()}.
  \item Retrieve all edges from Neo4j.
  \item Filter by cosine similarity to relation keywords.
  \item Run 20-iteration Personalised PageRank.
  \item Apply temporal decay and corroboration weighting.
  \item Detect and remove contradictions.
\end{enumerate}
\textbf{Output}: Ranked list of \texttt{KnowledgePath} objects.

\subsection{Module: EnhancedConflictResolver}

\textbf{Input}: \texttt{KnowledgePath} list, LLM, CFG\\
\textbf{Process}: For each path, compute $\Delta H = H_{\text{param}} - H_{\text{aug}}$
by sampling the LLM three times. Paths with $\Delta H < 0$ are excluded.
The remaining paths are assembled into the final prompt, and a confidence
score $\kappa$ is computed.\\
\textbf{Output}: Answer string, confidence score, conflict count, audit trail.

\section{System Sequence Diagram}

A typical user session breaks into two phases: corpus loading followed by
query execution. The steps below trace those phases in the order they actually
happen:

\begin{enumerate}
    \item User selects a corpus and clicks \emph{Load and Build Graph} in the web interface.
    \item \texttt{POST /api/load} triggers \texttt{EnhancedPipeline.ingest(docs)}:
    \begin{enumerate}
        \item \texttt{\_infer\_schema()} $\rightarrow$ Ollama $\rightarrow$ CFG updated
        \item \texttt{DETACH DELETE} $\rightarrow$ Neo4j cleared
        \item \texttt{LLMGraphTransformer} $\rightarrow$ Ollama $\rightarrow$ Neo4j
        \item \texttt{\_extract()} loop $\rightarrow$ Ollama $\rightarrow$ \texttt{\_accumulate()}
        \item \texttt{\_store\_all()} $\rightarrow$ Neo4j (MERGE writes)
        \item \texttt{\_disambiguate()} $\rightarrow$ Neo4j (APOC merge)
        \item \texttt{\_normalize()} $\rightarrow$ Ollama $\rightarrow$ Neo4j (APOC setType)
    \end{enumerate}
    \item User types a question; \texttt{POST /api/query/v5} triggers \texttt{EnhancedPipeline.query(q)}:
    \begin{enumerate}
        \item \texttt{\_keys()} $\rightarrow$ Ollama
        \item \texttt{detect\_intent()} $\rightarrow$ Ollama
        \item \texttt{\_edges()} $\rightarrow$ Neo4j
        \item \texttt{\_ppr()} (local NumPy computation)
        \item \texttt{\_detect\_contradictions()} (local)
        \item \texttt{\_entropy()} calls $\rightarrow$ Ollama
        \item \texttt{\_build\_chain()} (local)
        \item Final answer prompt $\rightarrow$ Ollama $\rightarrow$ answer string
        \item \texttt{\_confidence()} (local)
    \end{enumerate}
    \item JSON response returns to browser; confidence bar and path display rendered.
\end{enumerate}
