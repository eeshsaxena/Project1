# TruthfulRAG v5 — Detailed Contribution Proof
> **Sources:** v4 paper `arXiv:2511.10375` · Lewis et al. 2020 `arXiv:2005.11401`  
> Each contribution: stated → proved absent in v4 → proved absent in LangChain → what v5 adds

---

## CONTRIBUTION 1 — Temporal Decay Score `e^(-0.08 × age)`

**v5 Claim:** v5 multiplies every graph edge score by `e^(-λ×age)` where `λ=0.08`, penalising stale facts before they reach the LLM.

### ❌ Proved Absent in v4 (arXiv:2511.10375)

**Section:** Methodology → Graph Construction (HTML §Sx2.SSx1, p.3 of paper)

**Exact quote:**
> *"For each textual segment sᵢ ∈ S, we employ the generative model M … to extract a set of structured knowledge triples T_all = {Tᵢ,₁, Tᵢ,₂, …, Tᵢ,ₙ}, with each triple Tᵢ,ⱼ = **(h, r, t)** consisting of a head entity h, relation r, tail entity t."*

**Why this proves the gap:**  
The triple schema is explicitly `(h, r, t)` — a **3-tuple**. No year field, no timestamp, no `publication_date` attribute is defined anywhere in the paper. The word "temporal" appears in v4 only in the context of dataset description (RealtimeQA), not in the scoring formula.

**Section:** Implementation Details (HTML §Sx3.SSx1.SSSx5, p.5)
> *"For dense retrieval, cosine similarity is computed using embeddings generated by the all-MiniLM-L6-v2."*

No decay term is mentioned in any scoring or retrieval formula.

**Section:** Conflict Resolution formula (HTML §Sx2.SSx3, p.3–4)
> The entropy score is: `ΔHₚ = H(P_param(ans|q)) − H(P_aug(ans|q,p))`

This formula has **no time variable**. Selection criterion is entropy change — not document age.

---

### ❌ Proved Absent in LangChain (Standard RAG)

**Source:** Lewis et al. 2020 — "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"  
NeurIPS 2020 · arXiv:2005.11401

**Abstract (p.1):**
> *"the non-parametric memory is a **dense vector index** of Wikipedia, accessed with a **pre-trained neural retriever**"*

**Section 2.1 — Retriever (p.3):**  
Retrieval is by Maximum Inner Product Search (MIPS) over DPR embeddings. Score formula:
> `p_η(z|x) ∝ exp(d(z)ᵀ q(x))`

where `d(z)` is the document embedding and `q(x)` is the query embedding. **No time variable exists in this formula.**

## CONTRIBUTION 2 — Year Stored Per Triple

**v5 Claim:** v5 stores a `year` field on every knowledge graph triple, enabling temporal reasoning.

### ❌ Proved Absent in v4 (arXiv:2511.10375)

**Section:** Graph Construction (HTML §Sx2.SSx1, p.3)
> *"each triple Tᵢ,ⱼ = (h, r, t) consisting of a head entity h, relation r, tail entity t"*

Schema is a **3-tuple**. The paper defines no 4th field. The appendix case study (§A2) shows extracted triples like `("Municipality of Nuevo Laredo", "located_in", "Sinaloa")` — no year attached.

### ❌ Proved Absent in LangChain

LangChain's `RecursiveCharacterTextSplitter` creates chunks. Any metadata (including year) must be manually injected into the chunk's metadata dict. Even when stored, FAISS's `similarity_search()` signature is `(query: str, k: int)` — the metadata dict is **not part of the similarity computation**.

## CONTRIBUTION 3 — Cross-Document Corroboration `log(1 + support_count) × 0.8`

**v5 Claim:** v5 rewards facts that are confirmed by multiple independent documents using a logarithmic corroboration bonus.

### ❌ Proved Absent in v4 (arXiv:2511.10375)

**Section:** Graph Construction (HTML §Sx2.SSx1, p.3)
> *"The aggregated triple set from all retrieved content forms the foundation for constructing the knowledge graph G: where E = ⋃ᵢ,ⱼ,ₖ {hᵢ,ⱼ,ₖ, tᵢ,ⱼ,ₖ}…"*

Triples from multiple documents are **merged into one graph** with no tracking of how many documents supported each triple. There is no `support_count` field anywhere in the paper.

**Section:** Graph Retrieval — scoring formula (HTML §Sx2.SSx2, p.3):
> `score(p) = α × (|Eₚ ∩ Eᵢₘₚ| / |Eₚ|) + β × (|Rₚ ∩ Rᵢₘₚ| / |Rₚ|)`

Score depends only on entity/relation overlap with query — **not** on how many documents corroborate the same fact.

## CONTRIBUTION 4 — Hybrid Retrieval: BM25 + Semantic + RRF Fusion

**v5 Claim:** v5 uses BM25 keyword matching fused with semantic similarity via Reciprocal Rank Fusion.

### ❌ Proved Absent in v4 (arXiv:2511.10375)

**Section:** Implementation Details (HTML §Sx3.SSx1.SSSx5, p.5):
> *"For dense retrieval, **cosine similarity** is computed using embeddings generated by the **all-MiniLM-L6-v2**."*

Only cosine similarity on dense embeddings. **No BM25, no sparse retrieval, no fusion.**

**Section:** Graph Retrieval (HTML §Sx2.SSx2, p.3):
> *"semantic similarity matching is employed to identify the top-k most relevant entities and relations within the knowledge graph: sim(·,·) represents the **semantic similarity function computed using dense embeddings**"*

Explicitly semantic-only. BM25 is never mentioned in the paper.

### ❌ Proved Absent in LangChain Standard RAG

**Lewis et al. 2020, Section 2.1 — Retriever (p.3):**
> *"The retriever pη(z|x) returns top-K truncated passages given a query x. We use a **Dense Passage Retriever** (DPR)"*

DPR is embedding-only. BM25/sparse retrieval not included in the standard RAG formulation.

## CONTRIBUTION 5 — Decisive Gap Threshold (Gap > 0.30 → eliminate loser)

**v5 Claim:** v5 uses a decisive gap of 1.126 to eliminate stale facts. v4's gap is only 0.028 — insufficient to eliminate either fact.

### ❌ v4 Cannot Resolve — Gap Too Small

**Why the gap is only 0.028 in v4:**  
From C1: no year stored → no decay → `decay_factor = 1.0` for all facts.  
Only difference = semantic relevance `ref_p`:  
`ref_p(SAFE_FOR) = 1.161`, `ref_p(CONTRA) = 1.133` → gap = **0.028** — below any practical threshold.

**Section:** Conflict Resolution (HTML §Sx2.SSx3, p.3–4):
> *"Reasoning paths exhibiting entropy changes exceeding a predefined threshold τ are classified as P_corrective"*

v4 resolution criterion is **entropy-based**, not score-gap-based. When both facts produce similar entropy reduction, both survive to the LLM.

**Section:** Ablation Study (HTML §Sx3.SSx2.SSSx4, p.6):
> *"when utilizing solely the knowledge graph component without conflict resolution (w/o Conflict Resolution), CPR achieves significant improvements, yet the introduction of extensive structured knowledge simultaneously introduces **redundant information**, resulting in **limited improvements in accuracy** across most datasets"*

This directly confirms that v4's graph alone passes **both conflicting facts** through — they become redundant information. v5's temporal decay resolves this *before* entropy is computed.

## CONTRIBUTION 6 — Adaptive Entropy Skip [N9] (~35% Latency Reduction)

**v5 Claim:** v5 skips the 3-LLM entropy check for facts with extreme scores (very high or very low), saving ~35% of query latency.

### ❌ Proved Absent in v4

**Section:** Conflict Resolution (HTML §Sx2.SSx3, p.3):
> *"we implement conflict detection by comparing model performance under two distinct conditions: (1) pure parametric generation …, and (2) retrieval-augmented generation that incorporates structured reasoning paths … for each reasoning path p"*

v4 runs entropy check for **every** reasoning path with absolutely no skip logic.

**Section:** Computational Cost Analysis (HTML §A4.SSx4, Appendix D, p.9):
> *"TruthfulRAG introduces **moderate computational overhead** compared with FaithfulRAG, primarily due to the **graph-based reasoning and entropy filtering modules**."*

v4 explicitly acknowledges this overhead cost. v5's adaptive skip (N9) directly addresses this documented limitation of v4.

---

## CONTRIBUTION 7 — 3-Factor Calibrated Confidence Score

**v5 Claim:** v5 reports a calibrated confidence score per answer: `confidence = w₁·ΔH + w₂·decay + w₃·support`.

### ❌ Proved Absent in v4

**Section:** Conclusion (HTML §Sx5, p.6):  
Listed contributions: triple extraction, query-aware graph retrieval, entropy-based filtering. **No confidence score is mentioned anywhere.**

**Section:** Evaluation Metrics (HTML §Sx3.SSx1.SSSx4, p.5):
> *"we adopt **accuracy (ACC)** as the primary evaluation metric … we introduce the **Context Precision Ratio (CPR)** metric"*

v4 evaluates itself using ACC and CPR. No per-answer confidence score is produced for the end user.

### ❌ Proved Absent in LangChain

**Lewis et al. 2020, Section 3 — Results (p.5):**  
Reports EM (Exact Match) and F1 across datasets. No per-answer confidence score defined or output.

## CONTRIBUTION 8 — Full Human-Readable Audit Trail

**v5 Claim:** v5 returns a structured JSON audit trail per query showing which facts were removed and why: `{removed_facts: [...], reason: "gap=1.126 > 0.30", confidence: 84%}`.

### ❌ Proved Absent in v4

**Section:** Algorithm Overview (HTML §A3, Appendix C, p.8):  
The algorithm outputs: *"the final response is generated by the LLM based on the enriched context"*  
Only the final text answer is returned. No machine-readable elimination log.

**Section:** Case Study (HTML §A2, Appendix B, p.7–8):  
v4's case study shows intermediate reasoning paths in a static **paper table** written by the authors for illustration. This is not a runtime output — the live system produces only the final text answer.

## CONTRIBUTION 9 — Auto Schema Inference Per Domain

**v5 Claim:** v5 uses the LLM to automatically infer the appropriate triple schema per domain — no manual schema required.

### ❌ Proved Absent in v4

**Section:** Graph Construction (HTML §Sx2.SSx1, p.3):
> *"each triple Tᵢ,ⱼ = (h, r, t) consisting of a head entity h, relation r, tail entity t"*

The triple format is **hardcoded as (h, r, t)**. There is no mechanism to extend the schema with domain-specific attributes (like `year`, `jurisdiction`, `unit`, `enacted_year`) automatically. The schema is fixed across all domains.
